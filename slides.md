# バイブコーディングで露わになった課題

- AIに口頭で指示しながら実装を進めたが、命令が積み上がるほど文脈が崩壊
- PRレビューで仕様の抜け漏れが多数見つかり、最終的に人手で再実装する羽目に
- 「仕様を先に固める」「レビュー観点を残す」仕組みが必要だと痛感

---

# 立て直しのために整えた運用

- ブランチ作成直後に要件定義ファイルをコミットし、AIとの対話履歴とリンクさせる
- 要件定義→実装命令→レビュー観点をテンプレ化して共有
- git worktree で検証用ディレクトリを切り、進行中の実装を止めずに修正・テスト

---

# AIが仕様を書けるようになった転換点

- GPT-4.1 / Claude 4 で長文ドキュメント生成が安定し、要件定義のドラフトをAIに任せられるようになった
- 実装より先に「目的・制約・検証観点」をAIと対話し、エンジニアはレビューと修正指示に集中
- git worktree を組み合わせることで、進行中の実装を止めずに不具合調査や新機能検証を並列化できる環境が整った

> コードを書く手が減り、AIと向き合う時間がスキルの差を生み始めている

---

# モデル進化が後押しした精度

- GPT-4.1（2025/4/14 発表）で長文指示とドキュメント生成の安定性が向上
- Claude 4（2025/5/23 発表）で仕様記述からの要件整理力が改善
- これにより AIに要件定義→実装指示まで委ね、エンジニアがレビュー役として価値を出せる水準に到達

---

# エンジニアの役割シフト

- AIは定型コードや反復作業を肩代わりするが、価値を判断するのは人間
- 役割は「コードを書く人」から「AIに問いを投げ、レビューする人」へシフト
- U35世代は早期にこの役割を体験し、設計と品質判断を学ぶことでキャリアを伸ばせる

---

# AIに要件定義を書かせる流れ

1. 課題と成果物を整理し、AIへゴールと制約を伝える
2. 生成された要件定義・詳細設計をレビューし、補足や修正指示を出す
3. 実装命令→AI出力→差分レビュー→テストのサイクルで品質を確保
4. 得られた気づきを再びドキュメントへ反映し、ルールとして蓄積

  - AI出力をレビューし、テストやガイドラインに沿って統合できる

- git worktree などで進行中の実装を止めずに検証・修正を回せることが期待値

---

# git worktreeで止めない開発を支える

```bash
# 例: 作業ブランチごとに worktree を分離
git worktree add ../feature-a feature/a
git worktree add ../feature-b feature/b
```

- 稼働中ブランチとは別にワークツリーを切り、不具合調査や検証を中断なく進める
- 新機能用ワークツリーでも要件定義を先にコミットし、AIとのやり取りを記録
- Cursor の `rules/` ディレクトリをGitで共有し、複数タスクのガイドラインを同期

---

# チームで鍛えたいスキル

- **AIドキュメント活用力**: 課題を分解してAIに説明し、要件定義・詳細設計を引き出す力
- **設計レビュー力**: 生成された仕様を読み解き、リスクや抜け漏れを補正する力
- **実装監修力**: 出力コードをテスト・レビューし、要件に合わせて統合する力
- これらを育成テーマとしてドキュメントやレビュー習慣に落とし込み、若手評価の軸に据える

---

# AI時代のチーム運営で大切にしていること

- 並列開発をゴールに据えるのではなく、AIに意図を伝えレビューする力を評価軸にする
- 熟練者がレビューの型を示し、若手がAIとの対話と振り返りを繰り返す仕組みを整える
- 要件定義ドキュメントを読み解き、リスクを見つけて是正指示を出す力をチームで育てる

---

# まとめ

- バイブコーディングの手戻りを経て、要件定義を先に固めるドキュメント先行運用に舵を切った
- AIが仕様ドキュメントまで書けるようになり、エンジニアは設計とレビューに価値を移す段階に入った
- git worktree は進行中の実装を止めずに検証・修正を回す土台であり、要件定義と振り返りを支える
- U35世代が設計力と品質判断を磨くことが "Hello, new world!" を掴む近道

> まずは要件定義をAIに書かせ、git worktreeで結果を保存・比較する小さな実験から始めてみてください
